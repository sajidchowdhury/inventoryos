  1303‚Üí=== COST LEAKAGE RED FLAGS (10-item checklist) ===
  1304‚Üí
  1305‚Üí1. **Missing cache** ‚Äî endpoints that always call the AI even for repeat queries
  1306‚Üí   ‚ùå PRODUCT-ASSISTANT (all 4 sub-actions) ‚Äî explicitly no cache per route header. Every button click is a fresh LLM call. If a user generates a description for the same product 5 times, that's 5 LLM calls (rate-limited at 5/60s burst, so a 6th click within a minute is blocked). For a stable product (no name change), a 1-hour cache would eliminate ~90% of repeat calls. ‚öÝÔ∏è Medium severity ‚Äî defensible design choice for descriptions, but check_interactions (which is deterministic given the same med+condition list) really should be cached.
  1307‚Üí   ‚úÖ chat, insights, expiry-optimizer all have 24h cache.
  1308‚Üí   ‚úÖ forecast, reorder don't need cache (no LLM).
  1309‚Üí
  1310‚Üí2. **Missing rate limit** ‚Äî endpoints with no per-user/per-business cap
  1311‚Üí   ‚ùå FORECAST ‚Äî does not call checkAILimit() (because no LLM call). Heavy DB aggregations on 90-day window can be abused by rapid clicking, but no AI cost.
  1312‚Üí   ‚ùå REORDER ‚Äî same as forecast.
  1313‚Üí   ‚úÖ chat, insights, expiry-optimizer, product-assistant all enforce 4-tier rate limits.
  1314‚Üí
  1315‚Üí3. **Unbounded context** ‚Äî endpoints that send "all products" or "all sales" with no row cap
  1316‚Üí   ‚ùå EXPIRY-OPTIMIZER ‚Äî `db.batch.findMany` (line 66-84) has NO `take:` limit. Every expiring batch within 90 days is JSON-serialized and sent to GLM-4. Worst case: pharmacy with 500 expiring batches √ó ~250 chars each = ~125KB = ~36K tokens per call. At 0.03 BDT/1K tokens = ~1.08 BDT per single call. Across 50 daily calls = 54 BDT/day per business. HIGH SEVERITY.
  1317‚Üí   ‚ùå PRODUCT-ASSISTANT check_interactions ‚Äî `products` array from request body has NO length validation (line 124-129). Caller could submit 100+ medications. HIGH SEVERITY (but only triggered by intentional abuse or unusual use case).
  1318‚Üí   ‚öÝÔ∏è CHAT ‚Äî topStockProducts capped at 20, topSelling capped at 10, but aggregate counts are unbounded (12 separate Prisma aggregations). Total context ~1.4K tokens. LOW SEVERITY (capped sufficiently).
  1319‚Üí   ‚öÝÔ∏è INSIGHTS ‚Äî topProducts take:10, lowStockProducts take:10, expiringBatches take:10 ‚Äî all capped. LOW SEVERITY.
  1320‚Üí   ‚úÖ FORECAST ‚Äî fetches all 90-day saleItems but never sends to LLM. No AI cost.
  1321‚Üí   ‚úÖ REORDER ‚Äî same as forecast.
  1322‚Üí
  1323‚Üí4. **Large system prompts** ‚Äî system prompts over 500 tokens
  1324‚Üí   ‚ùå INSIGHTS ‚Äî system prompt ~600 tokens (lines 215-241, includes full JSON schema explanation). Combined with ~1,250-token dataSummary = ~1,850 input tokens per call. At max daily quota of 50 calls = 92,500 input tokens/day/business just for insights. MEDIUM SEVERITY.
  1325‚Üí   ‚öÝÔ∏è EXPIRY-OPTIMIZER ‚Äî ~450 tokens (borderline, under 500).
  1326‚Üí   ‚úÖ chat, product-assistant all under 300 tokens.
  1327‚Üí
  1328‚Üí5. **No max_tokens on output** ‚Äî endpoints that let the AI ramble
  1329‚Üí   ‚ùå ALL 4 LLM-CALLING ENDPOINTS (chat, insights, expiry-optimizer, product-assistant √ó 4 sub-actions). Every `zai.chat.completions.create({ messages, thinking: { type: "disabled" } })` call passes NO `max_tokens` / `max_output_tokens` parameter. GLM-4 has no enforced output cap.
  1330‚Üí   - Worst case: insights asks for "5-8 insights and 3-5 recommendations" with no length bound ‚Äî GLM-4 could return 5,000+ token JSON. At 0.03 BDT/1K = ~0.15 BDT per response just for output. Across 50 calls/day = 7.5 BDT/day/business.
  1331‚Üí   - Best fix: add `max_tokens: 1024` (chat) or `max_tokens: 2048` (insights, expiry-optimizer with large batch counts).
  1332‚Üí   HIGH SEVERITY ‚Äî easiest single fix to reduce cost variance.
  1333‚Üí
  1334‚Üí6. **Cron-triggered AI calls** ‚Äî any AI feature called automatically
  1335‚Üí   ‚úÖ NONE. Verified by reading all 3 cron routes (daily-maintenance, hourly-subscriptions, nightly-stats) and cron-jobs.ts. Cron jobs only:
  1336‚Üí     - Snapshot KPIs into BusinessDailyStats (reads AIUsageLog but does NOT call AI)
  1337‚Üí     - Auto-suspend expired paid subscriptions (disables aiEnabled when suspending pro_ai tier ‚Äî actually REDUCES AI cost)
  1338‚Üí     - Prune old logs/OTPs/Sessions/AIResponseCache entries
  1339‚Üí   No AI endpoint is invoked by any background job.
  1340‚Üí
  1341‚Üí7. **No usage logging** ‚Äî endpoints that don't write to AIUsageLog
  1342‚Üí   ‚ùå FORECAST ‚Äî does not call logAIUsage() (no LLM call, so technically correct, but the endpoint is named `ai/forecast` and presented to users as an AI feature ‚Üí super-admin dashboard undercounts "AI feature" usage).
  1343‚Üí   ‚ùå REORDER ‚Äî same as forecast.
  1344‚Üí   ‚úÖ chat, insights, expiry-optimizer, product-assistant all log every outcome (success, cache hit, SQL-router hit, rate-limit block, LLM failure) via logAIUsage().
  1345‚Üí
  1346‚Üí8. **No free-tier guard** ‚Äî endpoints that don't check subscription tier before calling
  1347‚Üí   ‚öÝÔ∏è PARTIAL: checkAILimit() checks `subscriptionStatus` (blocks suspended/cancelled) and `aiEnabled` (must be true), but does NOT check `subscriptionTier` directly. The `subscriptionStatus` defaults to "trial" ‚Äî which is ALLOWED. So:
  1348‚Üí     - A free-tier business (subscriptionTier="free") with `subscriptionStatus="trial"` and `aiEnabled=false` ‚Üí BLOCKED by aiEnabled check ‚úÖ
  1349‚Üí     - A free-tier business with `subscriptionStatus="trial"` and `aiEnabled=true` (manually toggled by an admin, or via a bug) ‚Üí ALLOWED ‚ùå
  1350‚Üí   The tier ladder (free/pro/pro_ai) is DEFINED in src/lib/feature-gate.ts (lines 70-160) but NO AI route handler imports or invokes `getTierConfig()` / `isFeatureEnabled()`.   1353‚Üí9. **Streaming endpoints** ‚Äî chat endpoints that may loop or retry on failure
  1354‚Üí   ‚öÝÔ∏è CHAT ‚Äî `zai.chat.completions.create` is called in NON-STREAMING mode (no `stream: true` option, line 257-260). The full response is awaited. NO retry logic in the route handler. On failure, buildFallback() is returned with `retryable: true` ‚Äî the UI (AIChat.tsx line 86-91) catches the error and shows a generic "Sorry, I couldn't process that" message ‚Äî NO automatic retry. ‚úÖ well-behaved.
  1355‚Üí   ‚öÝÔ∏è AIChat.tsx line 71 sends `history: messages.slice(-8)` ‚Äî meaning each turn re-sends up to 8 prior messages (~500 tokens of history) on every call. Over a long conversation this is a fixed ~500-token overhead per call. Not a leak per se, but a constant cost-per-turn.
  1356‚Üí   ‚úÖ No streaming. No retry loops. No exponential backoff that could multiply cost.

  1359‚Üí    ‚ùå REORDER ‚Äî ReorderSuggestions.tsx line 85: `useEffect(() => { fetchData(); }, [fetchData]);` fires on every component mount.   1363‚Üí=== ADDITIONAL FINDINGS ===
  1364‚Üí
  1365‚Üí- Cost model: `COST_PER_1K_TOKENS_BDT = 0.03` BDT per 1,000 tokens (src/lib/ai-rate-limit.ts line 32). At the default pro_ai monthly quota of 500,000 tokens, max monthly cost per business = 500,000 / 1,000 √ó 0.03 = 15 BDT/month (~$0.14 USD). At the 1,000 calls/month cap with avg ~2,000 input + ~500 output tokens per call = ~2.5M tokens/month = ~75 BDT/month (~$0.68 USD). The token budget (500K) is the binding constraint, not the call count (1,000).
  1366‚Üí- The costEstimate is computed from `tokensUsed` (which includes both input + output via `completion.usage.total_tokens` when reported by the SDK, or estimated via `text.length / 3.5` heuristic ‚Äî src/lib/ai-rate-limit.ts line 79-82). The estimate is rough.
  1367‚Üí- Token-estimate heuristic: `Math.ceil(text.length / 3.5)` (line 81) ‚Äî uses 3.5 chars/token. Industry standard for English is ~4 chars/token; for mixed English/Bangla (as in some user prompts), 3.5 is reasonable.
  1368‚Üí- The abuse-flag threshold (>20 calls/day = "high_usage", >40 = "possible_abuse") is BELOW the daily limit (50) ‚Äî so the super-admin sees abuse warnings BEFORE the rate limiter blocks the user. This is a reporting-only mechanism; no automatic throttling kicks in at the abuse threshold.
  1369‚Üí- AIResponseCache TTL: 24 hours (CACHE_TTL_HOURS = 24, src/lib/ai-cache.ts line 24). Lazy expiry on read + bulk prune in daily-maintenance cron. The dataHash mechanism means cache invalidates correctly whenever the underlying business data changes (new sale, new batch, stock change) ‚Äî so 24h TTL is safe.
  1370‚Üí- All 4 LLM routes pass `thinking: { type: "disabled" }` to zai.chat.completions.create ‚Äî disables GLM-4's "thinking" mode (which would add reasoning tokens). This is a cost-saving measure ‚úÖ.
  1371‚Üí- The chat route sends system prompt as `role: "assistant"` (line 249) ‚Äî unusual; most LLM SDKs use `role: "system"`. This works because the z-ai-web-dev-sdk's chat-completions endpoint accepts arbitrary role sequences, but it means the system instructions are NOT given special "system" precedence. Likely a minor behavioral quirk.
  1372‚Üí- Per-call token tracking is best-effort: routes check `completion.usage?.total_tokens` first (line 266-270 in chat, similar in others). If the SDK doesn't report usage (some response shapes), falls back to `estimateTokens(systemPrompt) + estimateTokens(message) + estimateTokens(aiResponse)`. This means usage stats are accurate when the SDK reports, heuristic when it doesn't.
